# PharmaOS — خارطة الطريق التطويرية التفصيلية

> خارطة طريق كاملة لبناء نظام إدارة الصيدلية PharmaOS من الصفر (offline أولاً، متعدد العملات، مصر أساسي، مطوّر واحد) — مراحل، معالم، معايير قبول، امتثال مصري، ومخاطر.

## ١. نظرة عامة تنفيذية

**PharmaOS** نظام إدارة صيدلية هجين (offline أولاً + مزامنة سحابية) يُبنى من الصفر بمستوى إنتاجي، متعدد الدول والعملات مع **مصر (EGP)** كسوق أساسي، وينفّذه **مطوّر واحد**.

تُقسّم الخطة البناء إلى **مرحلة تأسيس (Phase 0)** يعقبها أربع مراحل، مع **مسارات عرضية** ثابتة (أمان، أداء، اختبارات، عربية، نسخ احتياطي، مراقبة، نشر، توثيق)، و**قسم امتثال مصري** مبني على بحث موثّق (ETA وEDA).

**فلسفة التنفيذ — رشيقة ومتدرّجة:** نبدأ بـ *هيكل ماشٍ* ثم *MVP* يُشغَّل في **صيدلية واحدة حقيقية**، ثم نتوسّع. الأولوية للقيمة القابلة للتشغيل قبل اكتمال كل البنية.

**تقدير زمني واقعي (دوام كامل، مطوّر واحد):**
- Phase 0: ٢–٣ أشهر
- Phase 1 حتى تشغيل حقيقي: ٣–٥ أشهر
- الإجمالي عبر المراحل الأربع: **١٤–٢٢ شهراً**

> الـ Stack ثابت كما في CLAUDE.md ولا يتغيّر إلا بموافقة مسبقة.

## ٢. المبادئ والقيود الحاكمة

**الـ Stack (ثابت — بإصدارات v1.1 المحدّثة):** Next.js 16 (App Router) + TypeScript + React 19، Electron (أحدث خط مستقر ≥33)، shadcn/ui + Tailwind CSS 4، Zustand 5 + TanStack Query 5، React Hook Form + Zod، FastAPI ≥0.115 (Python 3.12) + SQLAlchemy 2.0 async + Pydantic v2، Celery 5 + Redis 7، PostgreSQL 17 محلي (Docker) + Supabase (نفس الإصدار الرئيسي)، JWT RS256، Node.js 22 LTS.

**سياسة الإصدارات (قرار مُلزم):** التوافق المتبادل أولاً، ثم الأحدث. كل الإصدارات مثبّتة (pinned) في lockfiles، التحقق من مصفوفة التوافق عند الإنشاء، والترقيات بموجات مُختبَرة بين المراحل فقط — لا ترقية major منفردة أثناء تسليم مرحلة. كل ترقية تُسجَّل في docs/versions.md.

**العملة (قرار مُلزم):** تصميم متعدد العملات بمعيار ISO 4217 — أي عملة عالمية مدعومة بنيوياً (جداول currencies/countries/tax_profiles). **الافتراضي الحالي: مصر EGP.** التوسع لاحقاً لأي دولة إعدادٌ لا إعادة بناء. كل المبالغ DECIMAL — لا FLOAT أبداً.

**قيود لا تتغيّر:**
- **offline أولاً:** القاعدة المحلية مصدر الحقيقة؛ كل عملية تُحفظ محلياً فوراً ثم تدخل outbox للمزامنة.
- **جاهزية المزامنة** في كل جدول عبر `sync_version` من اليوم الأول.
- **أهداف الأداء:** TTI < ٢ث، مسح باركود→عرض < 50ms، استجابة API محلياً < 200ms، بحث < 100ms، إقلاع < ٣ث.
- **الأعمدة الإلزامية:** جداول الكتالوج عالمية (بلا branch_id)، والجداول التشغيلية تحمل branch_id — كلاهما مع UUID PK وtimestamps وsoft delete وsync_version (تصنيف v1.1).
- **الدفعات مصدر الحقيقة الوحيد للكمية** — branch_inventory كاش مشتق قابل لإعادة البناء.

**القواعد المحظورة (v1.1 — أصبحت ٢٠):** الـ١٤ الأصلية + لا كمية على جدول الأدوية، لا تحديث كمية بلا stock_movement، لا FLOAT للمال، لا بيع من دفعة غير نشطة، لا ترقية major منفردة، لا إيصال ضريبي بلا توقيع صالح.

**قيد المورد:** مطوّر واحد ⇒ ترتيب أولويات صارم وتأجيل الأثقل.

> المرجع التفصيلي الكامل: **CLAUDE.md v1.1** — الوثيقتان متطابقتان الآن؛ عند أي تعارض مستقبلي CLAUDE.md يفوز ويُحدَّث هذا المستند.

## ٣. الهيكل الماشي (Milestone 0)

قبل التوسّع الأفقي في البنية، أنجز **شريحة رأسية رفيعة** تعبر كل الطبقات وتفكّك أصعب المخاطر مبكراً.

**المسار:** مسح باركود ← إضافة صنف للسلة ← إنشاء فاتورة ← طباعة 80mm ← حفظ محلي offline ← قيد في `stock_movements`.

**لماذا:** يثبت أن أصعب أجزاء المشروع (الأجهزة الطرفية + العمل بلا إنترنت + الطباعة) تعمل في **أول أسبوعين**، لا بعد أشهر — وهو أهم تقليل مخاطر لمطوّر واحد.

**معيار النجاح:** بيع صنف واحد فعلياً على جهاز حقيقي بلا إنترنت، طباعة إيصال حراري، وبقاء البيانات سليمة بعد إعادة تشغيل الجهاز.

## Phase 0 — التأسيس (Foundations)

**الأهداف:** monorepo يعمل بكل الأزمنة، مخطط قاعدة أساسي، عمود فقري للمصادقة والصلاحيات، CI، نسخ احتياطي وتشفير — *قبل* أي ميزة حقيقية.

**المهام:**
- **المونوريبو:** pnpm workspaces + Turborepo؛ `apps/{desktop, web, api}` و`packages/{ui, db, shared}` بالبنية المحددة.
- **الأدوات:** TypeScript strict، ESLint، Prettier، Ruff/Black، Husky + commitlint (فرض صيغة الـ commits الواردة في الملف).
- **البنية المحلية:** Docker Compose (PostgreSQL 16، Redis 7).
- **Supabase:** إنشاء المشروع + سير عمل *migrations فقط* عبر Supabase CLI؛ تُطبَّق نفس migrations على Postgres المحلي (مصدر مخطط واحد)؛ لا تعديل مباشر لـ `/supabase`.
- **المخطط الأساسي:** الأعمدة الإلزامية على كل جدول + جداول Core (`branches, users, roles, permissions, role_permissions`) + تفعيل امتداد `pgcrypto`.
- **المصادقة:** JWT RS256 (زوج مفاتيح)، access 15د / refresh 7ي، `token_version`، سياسة كلمة المرور (argon2/bcrypt، ≥٨ محارف)، قفل بعد ٥ محاولات ١٥ دقيقة، httpOnly cookies + CSRF.
- **RBAC:** مصفوفة الصلاحيات من الملف؛ `require_permission` في كل endpoint + `hasPermission` guard في الواجهة (مع فحص backend دائماً).
- **نظام التصميم:** ألوان الملف + خطوط Cairo / IBM Plex Arabic + Inter للأرقام + RTL كامل + shadcn/ui.
- **النسخ الاحتياطي والتشفير (مسحوبان مبكراً):** `pg_dump` يومي مشفّر + **اختبار استعادة**، ونسخ سحابي **أحادي الاتجاه** لتفادي فقد بيانات الجهاز، وAES-256 للحقول الحساسة (إدارة المفتاح عبر `ENCRYPTION_KEY`).
- **متعدد العملات:** تجريد «ملف دولة» (عملة، نسبة ض.ق.م، قالب فاتورة، تقويم) بقيمة EGP افتراضية.
- **CI:** GitHub Actions — lint، type-check، unit tests، build، فحص migration up/down.

**المُخرَجات:** monorepo يقلع، migrations تعمل، تسجيل دخول فعّال، CI أخضر، نسخ/استعادة مُختبَرة.

**معايير القبول:** نسخة نظيفة → `docker compose up` → migrate → دخول `super_admin`؛ CI ينجح؛ استعادة من نسخة احتياطية تنجح؛ الإقلاع < ٣ث.

**التبعيات:** لا شيء (الأساس).
**التقدير (فرد، دوام كامل):** ٢–٣ أشهر.

## Phase 1 — النواة MVP (Core)

**الأهداف:** صيدلية حقيقية تبيع **offline** عبر POS بالباركود، تطبع الفواتير، وتدير المخزون وجلسات الكاشير.

**المهام:**
- **المصادقة والمستخدمون (واجهة):** CRUD، إسناد أدوار، تفعيل/تعطيل، إعادة تعيين كلمة المرور.
- **إعداد الفروع.**
- **كتالوج الأدوية:** CRUD + باركود **1D و2D (DataMatrix)** عبر جدول `medication_barcodes` + بحث عربي بإعداد `arabic_simple` المخصص ودالة التطبيع (الهمزات/الألف/التاء/التطويل/التشكيل) — **بذر أولي** من المجموعة المفتوحة CC0 (٢٤٬٨٦٨ صنفاً) + استيراد Excel.
- **⭐ تسلسل الوحدات (v1.1 — شرط تشغيل مصري):** نمذجة **علبة/شريط/قرص** عبر `medication_packaging` بسعر وباركود لكل مستوى، والبيع بأي مستوى من POS (اختصار F7 للتبديل)، وكل الكميات داخلياً بالوحدة الأصغر.
- **المخزون:** استلام بالدفعات، `medication_batches` مصدر الحقيقة، `stock_movements` لكل تغيير، **FEFO**، كاش `branch_inventory` مع فحص drift.
- **POS:** مسح باركود (< 50ms)، اختصارات (F2 بحث، F3 خصم، F4 إتمام، F5 طباعة، F6 مرتجع، F7 مستوى الوحدة، F8 درج، ESC)، تشغيل بلا ماوس.
- **الفاتورة والطباعة:** إيصال حراري 80mm عبر **ESC/POS مباشرة من Electron main process** (وفتح الدرج بأمر pulse) — لا اعتماد على طباعة المتصفح؛ والتقارير A4/A5 عبر WeasyPrint.
- **جلسات الكاشير:** فتح/إغلاق، عدّ النقدية، كشف الفروق.
- **Audit Log** موصول منذ أول عملية كتابة (append-only بقيد قاعدة).
- **اختبار الأجهزة الطرفية فعلياً:** طابعة حرارية، ماسح HID **ثنائي الأبعاد 2D حصراً** (جاهزية تتبّع الدواء)، درج نقود.

**المُخرَجات:** بيع كامل offline بأي مستوى وحدة، إيصال مطبوع + درج يفتح، خصم من الدفعات FEFO، جلسة كاشير، قيود تدقيق.

**معايير القبول:** بيع ١٠٠ صنف بالباركود offline بلا ماوس (منها بيع شريط وقرص مفرد)؛ طباعة < ١ث + فتح الدرج فعلياً؛ مسح→عرض < 50ms؛ اختبار crash recovery أثناء بيع؛ drift check بعد ١٠٠٠ عملية = صفر انحراف؛ test suite للصلاحيات كاملاً.

**التبعيات:** Phase 0.
**التقدير:** ٣–٥ أشهر. **نقطة إعادة معايرة:** بعد أول أسبوعين من POS قِس السرعة الفعلية وعدّل تقديرات ما بعدها.

## Phase 2 — الأعمال والامتثال المصري

**الأهداف:** تشغيل صيدلية كامل + **امتثال قانوني** (ETA + EDA Track & Trace) + ض.ق.م.

**المهام — الأعمال:**
- الموردون (كامل)، أوامر الشراء (طلب → موافقة → استلام)، الاستلام بمسح **2D** لكل عبوة.
- **تعميق تتبّع الدفعات** (الأساسيات في Phase 1): تنبيهات انتهاء ٣٠/٦٠/٩٠، **حجر تلقائي** للمنتهي (batch.status = quarantined)، تقارير دفعات، والتقاط التسلسل ورقم التشغيلة في `pack_serials` عند الاستلام والصرف.
- العملاء والولاء.
- الوصفات + **سجل منفصل للمواد الخاضعة للرقابة** (append-only) + تنبيهات.
- المرتجعات عبر **إشعارات دائنة** (لا تعديل للفاتورة المكتملة).
- المصروفات وتصنيفاتها.
- **ض.ق.م** حسب `tax_profiles` (بنية متعددة الدول)، وإظهارها في الفاتورة.

**المهام — الامتثال (مبني على بحث موثّق):**
- **وحدة الإيصال الإلكتروني (ETA):** بعد اكتمال المسار الورقي الموازي (انظر قسم الامتثال) — OAuth2 (`POST /connect/token`)، بناء الإيصال JSON موقّعاً بشهادة الختم X.509، الإرسال عبر `POST /api/v1/receiptsubmissions`، QR على الإيصال، ومعالجة الردود — كله عبر طابور `ereceipt_queue` بنمط outbox: **البيع لا يتوقف على الإنترنت أبداً**.
- **وحدة تتبّع الدواء (EDA T&T — قرارات ١٦١/٤٧٥/٨٠٤ لسنة ٢٠٢٥):** تسجيل المنشأة، إبلاغ أحداث الاستلام/الصرف/الإتلاف عبر طابور `tt_events`، واستيراد ما سُجّل بالحل المؤقت قبل الإطلاق (لا فجوة بيانات).

**المُخرَجات:** إيصال إلكتروني مقبول على بيئة اختبار ETA، التقاط وإبلاغ التسلسل، مشتريات/مرتجعات/عملاء/مصروفات كاملة.

**معايير القبول:** إيصال صالح مقبول على بيئة اختبار ETA؛ انقطاع إنترنت ٢٤ ساعة ← يتجمع الطابور ثم يُصرَّف كاملاً بلا فقد؛ مسح عبوة 2D ← التقاط التسلسل/التشغيلة/الانتهاء ← حدث مُبلَّغ؛ فرض FEFO ومنع بيع المحجور (E-STK-002)؛ مسار الإشعار الدائن كامل؛ سجل المواد الخاضعة للرقابة append-only.

**التبعيات:** Phase 1 (POS، الفواتير، الدفعات الأساسية) + اكتمال الأوراق الحكومية (المسار الموازي).
**التقدير:** ٤–٦ أشهر (الامتثال ثقيل). **نقطة إعادة معايرة:** بعد قبول أول إيصال على بيئة الاختبار — راجع المتبقي.

## Phase 3 — التحليلات (Analytics)

**الأهداف:** تقارير بجودة القرار + محرك تنبيهات ذكي + إشعارات.

**المهام:**
- تقارير المبيعات (يومي/شهري/سنوي)، والمخزون (مستوى، حركة)، والمالية.
- **محرك التنبيهات:** مخزون منخفض، نفاد، انتهاء (٣٠/٦٠/٩٠ يوماً)، خصم يتجاوز الحد (يتطلب موافقة)، فرق صندوق، فشل مزامنة، نسخ احتياطي متأخر.
- تحليل الأرباح والخسائر، أداء الموردين، سجل مشتريات العملاء.
- **نظام الإشعارات:** داخل التطبيق / سطح المكتب / بريد.

**معايير القبول:** تقرير يومي < ٣ث؛ إطلاق التنبيهات وفق القواعد؛ تسليم الإشعارات.

**التبعيات:** بيانات Phase 1–2.
**التقدير:** ٢–٣ أشهر.

## Phase 4 — المؤسسية (Enterprise)

**الأهداف:** التوسّع لفروع متعددة + مزامنة سحابية كاملة + جوال + تكاملات.

**المهام:**
- **إدارة الفروع المتعددة** (الكتالوج مشترك عالمياً منذ v1.1 — يسهّل هذه المرحلة جوهرياً).
- **محرك المزامنة السحابية الكامل (v1.1 — بلا CRDT):** Outbox Pattern + **Last-Write-Wins** حسب updated_at وsync_version + **قائمة مراجعة يدوية** لتعارضات الكيانات الحرجة (فواتير، مدفوعات، جلسات كاشير، مواد خاضعة للرقابة، تسلسلات العبوات)، عامل خلفي، ومؤشر حالة مزامنة دائم. *قرار معماري نهائي: حُذف CRDT — نمط الصيدلية (جهاز رئيسي لكل فرع + dashboard قراءة غالباً) لا يبرر تعقيده لمطوّر واحد.*
- **تطبيق الجوال (React Native)** — يبقى آخر البنود؛ يُقيَّم مقابل ويب متجاوب عند الوصول إليه.
- **تكامل التأمين** و**تكامل وزارة الصحة**.
- توسيع الامتثال والعملات لدول إضافية (SAR/ZATCA، AED، ...) — إعداد ملفات دول جديدة فوق البنية متعددة العملات القائمة.

**معايير القبول:** جهازان يتزامنان بلا فقد/تكرار؛ تعارض مفتعل على فاتورة يدخل قائمة المراجعة اليدوية ولا يُحسم صامتاً؛ توفيق offline→online كامل؛ حالة المزامنة مرئية دائماً.

**التبعيات:** كل ما سبق (وجاهزية sync_version من Phase 0).
**التقدير:** ٣–٦+ أشهر. **نقطة إعادة معايرة:** قبل بدء المرحلة راجع الحاجة الفعلية لكل بند — قد يكفي النسخ السحابي الأحادي + dashboard قراءة لسيناريو الصيدلية الواحدة.

## المسارات العرضية (Cross-cutting)

حاضرة في **كل** مرحلة، لا كمراحل منفصلة:

- **الأمان:** RBAC في كل مكان، Audit Log، AES-256، parameterized queries، rate limiting (دخول ٥/د، API ٢٠٠/د)، security headers، إبطال الجلسات عند تغيير كلمة المرور، **بوابة OWASP Top 10** قبل الإنتاج.
- **الأداء:** فهارس على كل الجداول، pagination إلزامي (≤١٠٠)، `EXPLAIN ANALYZE` لأي استعلام > 50ms، والتحقق من الأهداف كل مرحلة.
- **الجودة/الاختبار:** تغطية ٨٠٪ لمنطق الأعمال، اختبار تكامل لكل endpoint، E2E للمسارات الحرجة (دخول، بيع، طباعة، مرتجع)، **تركيز على صحة العمل offline وموثوقية الطباعة**، واختبار migrations up/down.
- **العربية/RTL:** RTL كامل، خطوط عربية، `tabular-nums`، تنسيق EGP، تقويم ميلادي/هجري.
- **النسخ الاحتياطي/التعافي:** يومي مشفّر + سحابي أحادي + تدريبات استعادة.
- **المراقبة:** مستويات السجلّات (ERROR/WARN/INFO/DEBUG)، تتبّع أخطاء، فحوص صحة، وزر «تواصل مع الدعم» يفتح تذكرة بالسياق.
- **النشر/DevOps:** توقيع Electron + تحديث تلقائي + مثبّتات، بيئتا staging وproduction للسحابة، انضباط migrations.
- **التوثيق/التدريب:** دليل مستخدم بالعربية، توثيق إداري، تدريب الطاقم، قاعدة معرفة.

### أ) الإيصال الإلكتروني (ETA) — B2C، ينطبق على الصيدليات

- منفصل عن الفاتورة الإلكترونية (B2B). الإلزام **بقوائم اسمية** عبر قرارات وزارية (القطاع الصحي مُلزَم منذ ١٥ أبريل ٢٠٢٣ بالقاهرة الكبرى والإسكندرية؛ المراحل مستمرة — المرحلة ٨ في يوليو/سبتمبر ٢٠٢٥).
- **إجراء أول:** التحقق من إلزام الصيدلية عبر `eta.gov.eg/ar/ereceipt-inquiry` برقم التسجيل الضريبي.
- **التكامل التقني:** تسجيل الملف الرقمي والأجهزة → بيانات API لكل جهاز POS؛ شهادة **ختم إلكتروني X.509**؛ **OAuth2** (`POST /connect/token`)؛ إرسال **JSON موقّع** عبر `POST /api/v1/receiptsubmissions`؛ **QR** على الإيصال؛ TLS 1.2+؛ طابور offline — البيع لا يتوقف على الإنترنت. SDK: `sdk.invoicing.eta.gov.eg` | دعم: 16395.

**📋 المسار الورقي الموازي (مسؤولية مالك الصيدلية — يبدأ فور اعتماد الخطة، لا ينتظر التطوير):**

| الخطوة | الجهة | ملاحظة |
| --- | --- | --- |
| ١. الاستعلام عن الإلزام برقم التسجيل | أداة استعلام ETA | فوراً |
| ٢. التسجيل بمنظومة الإيصال الإلكتروني | بوابة ETA / المأمورية | مستندات: تسجيل ضريبي، بطاقة، تفويض |
| ٣. استخراج شهادة الختم الإلكتروني X.509 | مقدم خدمة معتمد | قد تستغرق أسابيع — ابدأ مبكراً |
| ٤. تسجيل أجهزة POS بالمنظومة | بوابة ETA | يمنح بيانات API لكل جهاز |
| ٥. التسجيل ببرنامج التحفيز (فاتورتك) | بوابة ETA | مطلوب بقرارات الإلزام |

### ب) تتبّع الدواء (EDA Track & Trace) — 🔴 عاجل

- **قرار ١٦١/٢٠٢٥:** ترميز موحّد إلزامي **2D (GS1 DataMatrix)**: GTIN + تاريخ الانتهاء + رقم التشغيلة + رقم تسلسلي عشوائي.
- **قرار ٤٧٥/٢٠٢٥:** المنظومة القومية الموحّدة؛ كل عبوة تُتتبّع حتى المريض؛ الصيدليات ضمن المنشآت الملزمة (تسجيل + ماسحات + برمجيات متصلة).
- **قرار ٨٠٤/٢٠٢٥ (الدليل التنظيمي):** يُعرّف **«المنتج غير الممتثل»** (كود غير سليم / تسلسل مكرر / حدث غير مسجل بالمنصة) ويجيز **وقف تداوله وحجزه وإتلافه** — أي أن الفشل التقني في الإبلاغ يجعل منتجاً سليماً «غير قانوني».
- **المواعيد:** ١ فبراير ٢٠٢٦ للمستورد تام الصنع (**سارٍ**)، و**١ أغسطس ٢٠٢٦ للمحلي (وشيك)**. المنتجات المتداولة قبل الموعد تستمر حتى نفادها/انتهائها.
- **الأثر على PharmaOS:** ماسح **2D حصراً** من Phase 1، `pack_serials` + `tt_events` في المخطط من البداية، الاستلام والصرف يلتقطان التسلسل، ووحدة الإبلاغ القومي في Phase 2 بنمط outbox.

**⏳ الامتثال المؤقت (قبل جاهزية وحدة PharmaOS) — توصية عملية:**

1. **نافذة سماح واقعية:** مخزون الصيدلية المشترى قبل المواعيد يتداول قانونياً حتى نفاده/انتهائه — الضغط الفوري أقل مما يبدو، لكنه يتصاعد مع كل توريد جديد بعد ١ أغسطس.
2. المصادر العامة **لا تذكر تطبيقاً رسمياً جاهزاً للصيدليات** — المطلوب تحقق مباشر: التواصل مع **EDA** (البوابة/الخط الساخن) و**نقابة الصيادلة** عن قناة الإبلاغ المعتمدة للصيدليات (بوابة ويب، تطبيق، أو مزود معتمد) والالتزام بها مؤقتاً.
3. **التسجيل بالمنظومة القومية فوراً** حتى قبل جاهزية البرمجيات — التسجيل التزام مستقل عن التكامل، وقد يكشف قناة الإبلاغ الرسمية.
4. شراء **ماسحات 2D** من الآن — مطلوبة للامتثال المؤقت واللاحق معاً.
5. تصميم وحدة PharmaOS للتتبّع بحيث **تستورد** ما سُجّل يدوياً/مؤقتاً عند الإطلاق (لا فجوة بيانات).

### ج) مصدر قاعدة الأدوية

- **لا API رسمي** من EDA؛ أداة بحث ويب فقط (EDDB).
- **الأساس المجاني:** مجموعة CC0 (`karem505/egyptian-drug-database`) — ٢٤٬٨٦٨ صنفاً (أسماء عربية/إنجليزية، مادة فعّالة، مصنّع، سعر EGP) بصيغتي CSV/JSON. **بلا باركود** — يُستكمل من مزوّد تجاري أو تراكمياً بالمسح عند الاستلام.
- **الترقية عند الحاجة:** مزوّد تجاري (اشتراك شهري) للباركود والتحديث اليومي؛ التحقق الرسمي يبقى عبر EDA.

### د) التسعير والمواد الخاضعة للرقابة

- أسعار الأدوية **مقرَّرة حكومياً** ⇒ منطق تسعير يتبع التحديثات + سجل تاريخ أسعار لكل تغيير.
- سجل منفصل ومدقَّق للمواد الخاضعة للرقابة (append-only، بلا حذف نهائي أبداً).

## حدّ الـ MVP والمسار الرشيق

**داخل MVP (يوم التشغيل الأول لصيدلية واحدة):** مصادقة، فرع واحد، كتالوج أدوية مبذور، POS بالباركود (1D+2D)، بيع offline، طباعة 80mm، إدخال/إخراج مخزون، جلسة كاشير، نسخ احتياطي، تدقيق أساسي.

**مؤجَّل بعد MVP:** عمق المشتريات والموردين، الولاء، الوصفات، المرتجعات المتقدمة، التحليلات، الفروع المتعددة، المزامنة، الجوال.

**تنبيه بخصوص الامتثال:** الإيصال الإلكتروني (ETA) وتتبّع الدواء (EDA) **متطلبان قانونيان** وليسا اختياريين؛ فإن كانت الصيدلية مُلزَمة بالفعل، تُرفع أولويتهما فوق ترتيب MVP المعتاد ويُدمجان مبكراً. الـ MVP هنا يخص *التشغيل الداخلي*؛ الامتثال مسار قانوني موازٍ.

**المسار الرشيق:** M0 هيكل ماشٍ ← Phase 0 ← Phase 1 وتشغيل صيدلية تجريبية واحدة ← جمع ملاحظات واقعية ← Phase 2 والامتثال ← التوسّع.
